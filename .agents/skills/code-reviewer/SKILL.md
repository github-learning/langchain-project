# Code Reviewer - 代码审查专家技能

## 语言规范

- **交流与思考**：与用户交流、思考分析过程、解释方案时**必须使用中文**
- **审查报告**：中文编写，详细说明问题和修复建议
- **问题解释**：中文说明缺陷根因和影响范围

## 触发场景

- 用户提到「代码审查」「code review」「代码优化」「检查代码」等关键词
- 提交 PR / 代码片段需要审查时

## 核心审查维度（按优先级排序）

### 1. 功能正确性
- 检查代码是否实现预期功能
- 是否存在逻辑 bug
- 边界条件处理是否得当
- 异常情况是否遗漏

### 2. 安全合规
- 检查是否存在 OWASP Top10 安全漏洞
- 敏感信息泄露
- 权限控制是否得当
- 注入风险

### 3. 性能优化
- 循环冗余
- 内存泄漏
- 慢查询
- 资源未释放
- 重复计算

### 4. 代码质量
- 是否符合项目编码规范
- 命名是否合理
- 注释是否完整
- 是否存在重复代码、坏味道

### 5. 可维护性
- 是否遵循单一职责原则
- 是否有合理的抽象
- 是否易于扩展
- 是否有完整的测试覆盖

## 审查工作流

```
代码理解 → 逐维度审查 → 问题标注 → 修复方案 → 审查报告
    ↓          ↓           ↓          ↓          ↓
  理解上下文   5个维度    严重等级    具体代码    优先级排序
```

### 步骤1: 先理解代码的业务场景
- 预期功能是什么
- 上下文依赖有哪些
- 在项目中的位置

### 步骤2: 按5个维度逐行审查
- 标记每个发现的问题
- 记录问题位置和行号

### 步骤3: 对每个问题标注
- 严重等级
- 影响范围
- 根因分析

### 步骤4: 给出具体的修复代码
- 可直接复制使用的修复代码
- 明确标注修改的位置

### 步骤5: 生成结构化审查报告
- 优先展示高严重等级问题
- 给出整体优化建议

## 严重等级定义

| 等级 | 说明 | 处理要求 |
|------|------|---------|
| **Blocker** | 阻塞代码合并，必须修复 | 功能错误、安全漏洞、数据丢失风险 |
| **Critical** | 严重影响代码质量，强烈建议修复 | 性能瓶颈、严重的逻辑缺陷 |
| **Major** | 影响可维护性，建议修复 | 坏味道、重复代码、不规范的命名 |
| **Minor** | 不影响功能，优化项 | 注释不完整、格式不规范 |
| **Trivial** | 细节优化，可选修复 | 微小的改进建议 |

## 禁忌

- **不做无意义的格式挑错**，聚焦核心问题
- **不给出空泛的建议**，所有建议必须有具体的修复方案
- **不改变代码的核心业务逻辑**，仅优化实现方式
- **不忽略项目现有的代码规范**，必须与项目风格保持一致

## ontology_next 接口审查要点

### 1. 关键流程审查

#### 1.1 多意图拆分时上下文传递

**审查重点**：检查 `pre_extracted_entities` 是否正确传递给子意图。

```python
# 审查位置：chat_service.py 构建 SubIntent 时
sub_intents.append(
    SubIntent(
        sub_query=sq,
        pre_extracted_entities=pre_extracted_entities.to_dict()  # ← 必检
    )
)

# 审查位置：intent_processor.py 使用时
pre_extracted_entities=sub_intent.pre_extracted_entities  # ← 必检
```

**常见问题**：
- `pre_extracted_entities = None` 在 if 块外覆盖了之前的值
- SubIntent 未注入 entities
- intent_processor 未使用 sub_intent 的 entities

#### 1.2 内置问题开关

**审查重点**：检查 `enabled` 开关是否动态检查。

```python
# 正确：每次 match 重新检查
def match(self, query: str) -> dict:
    if not self._check_enabled():  # ← 动态检查
        return None

# 错误：只在初始化时检查一次
def match(self, query: str) -> dict:
    if not self._enabled:  # ← 可能不生效
        return None
```

### 2. 配置热更新审查

| 配置 | 文件 | 热更新 | 审查要点 |
|------|------|--------|----------|
| 内置问题 | `built_in_questions.json` | ✅ | `enabled` 开关动态检查 |
| 意图关键词 | `intent_keywords.json` | ✅ | mtime 检测机制 |
| 术语映射 | `industry_glossary.json` | ❌ | 需标注需重启 |

### 3. 回归测试场景（必检）

| 场景 | 输入 | 预期结果 | 审查问题 |
|------|------|----------|----------|
| 多意图-上下文中 | "牛仔的机针和转速呢" | 两个子查询都关联牛仔 | entities 是否传递 |
| 内置问题-启用 | "什么是服装的缝份" | LLM直接回答 | enabled 检查 |
| 单意图-面料类 | "牛仔用什么机针" | 返回牛仔面料参数 | 正常流程 |
| 统计查询 | "Error系列有哪些" | 返回错误列表 | 知识库查询 |

### 4. 错误处理审查

**必检错误码**：
- `ON_040001`: 意图处理超时
- `ON_050001`: LLM服务调用失败
- `ON_020001`: 意图未识别

### 5. 日志完整性审查

**必含日志点**：
- ✅ rewrite_query 前后的查询变化
- ✅ entity 提取结果
- ✅ 多意图拆分结果
- ✅ 并行处理开始/完成
- ⚠️ 禁止：敏感信息（密码、token）
