# TDD Workflow - 测试驱动开发技能

## 语言规范

- **交流与思考**：与用户交流、思考分析过程、解释方案时**必须使用中文**
- **测试说明**：中文描述测试用例的设计思路
- **重构解释**：中文说明重构原因和改进方向

## 触发场景

- 用户提到「TDD」「测试驱动开发」「单元测试」「写测试用例」「测试覆盖率」等关键词
- 需要先写测试再实现功能时

## 核心原则

- **先写测试，再写实现**：没有测试用例，不编写任何业务代码
- **最小实现原则**：仅编写刚好能让测试通过的代码，不提前实现额外功能
- **测试覆盖全面**：必须覆盖正常流程、边界条件、异常场景、错误处理
- **重构不改变功能**：重构阶段仅优化代码结构，不改变代码的对外行为

## 标准工作流

### RED → GREEN → REFACTOR

```
┌─────────────────────────────────────────────────────────────┐
│                    步骤1: 需求拆解与接口定义                        │
│  (红)                                                        │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │ 1. 确认需求的验收标准，拆解为可测试的功能点                        │ │
│  │ 2. 定义功能的输入、输出、接口、异常类型                          │ │
│  │ 3. 不编写任何业务实现代码，仅定义接口和类型                        │ │
│  └─────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                    步骤2: 编写失败的测试用例（RED）                     │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │ 1. 基于接口定义，编写对应的测试用例                            │ │
│  │ 2. 测试用例必须包含：                                       │ │
│  │    - 正常场景                                              │ │
│  │    - 边界条件                                              │ │
│  │    - 异常输入                                              │ │
│  │    - 错误处理场景                                          │ │
│  │ 3. 执行测试，确认测试全部失败（红）                          │ │
│  └─────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                    步骤3: 实现最小功能代码（GREEN）                   │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │ 1. 编写刚好能让所有测试通过的最小实现代码                        │ │
│  │ 2. 不提前优化、不添加额外功能                                 │ │
│  │ 3. 执行测试，确认所有测试全部通过（绿）                         │ │
│  └─────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                    步骤4: 重构优化（REFACTOR）                      │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │ 1. 在不改变代码对外行为、不影响测试通过的前提下优化代码               │ │
│  │ 2. 优化方向：可读性、可维护性、性能、消除重复代码、简化逻辑           │ │
│  │ 3. 每一步重构后，立即执行测试，确保所有测试依然通过                 │ │
│  └─────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                    步骤5: 验收与交付                               │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │ 1. 生成测试覆盖率报告，确保核心业务逻辑覆盖率≥80%                  │ │
│  │ 2. 整理测试用例、实现代码、重构说明                            │ │
│  │ 3. 给出后续的测试扩展建议                                    │ │
│  └─────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────┘
```

## 测试用例编写规范

### 测试用例结构

```python
import pytest
from app.services.example import IntentClassifier

class TestIntentClassifier:
    """意图分类器测试用例"""

    def test_normal_query_returns_correct_intent(self):
        """正常查询应返回正确意图"""
        classifier = IntentClassifier()
        result = classifier.classify("E01报警怎么解决")
        assert result.intent == "ElectricalFault"

    def test_empty_query_raises_value_error(self):
        """空查询应抛出ValueError"""
        classifier = IntentClassifier()
        with pytest.raises(ValueError, match="query不能为空"):
            classifier.classify("")

    def test_boundary_empty_string(self):
        """边界条件: 空字符串"""
        pass

    def test_boundary_whitespace_only(self):
        """边界条件: 仅包含空格"""
        pass

    def test_boundary_very_long_string(self):
        """边界条件: 超长字符串 (10000+字符)"""
        pass

    def test_boundary_special_characters(self):
        """边界条件: 特殊字符"""
        pass

    def test_error_handling_invalid_input_type(self):
        """错误处理: 无效输入类型"""
        classifier = IntentClassifier()
        with pytest.raises(TypeError):
            classifier.classify(123)
```

### 必须覆盖的边界条件

| 边界类型 | 示例 | 测试要点 |
|----------|------|----------|
| 空字符串 | `""` | 抛出特定异常 |
| 仅空格 | `"   "` | 正确处理 |
| 超长字符串 | `"a" * 10000` | 性能/截断 |
| 特殊字符 | `"!@#$%^&*()"` | 编码处理 |
| None 值 | `None` | 类型检查 |
| 数字输入 | `123` | 类型校验 |
| 空列表 | `[]` | 边界处理 |
| 注入攻击 | `"'; DROP TABLE"` | 安全处理 |

## 测试覆盖率要求

| 模块类型 | 最低覆盖率 | 核心功能覆盖率 |
|----------|------------|----------------|
| 核心业务逻辑 | 80% | 95% |
| 工具类/Helper | 70% | 90% |
| API 路由 | 90% | 100% |
| 数据模型 | 80% | 100% |

## 验收标准

每次 TDD 循环完成时，必须满足：

1. **所有测试通过**（Green）
2. **核心覆盖率达标**（≥80%）
3. **无警告或 error**（pytest -W error）
4. **重构后仍通过**（Refactor）

## 与其他角色协作

### 配合 Developer
- 提供测试用例模板
- 解释测试覆盖要求
- 协助编写复杂测试场景

### 配合 QA Engineer
- 提供单元测试报告
- 说明覆盖率情况
- 协助定位测试遗漏

## 常见问题

### Q: 测试太难写怎么办？
A: 说明接口设计可能有问题，先优化接口设计

### Q: 覆盖率达不到80%？
A: 识别未覆盖的分支，补充边界条件测试

### Q: 测试太慢影响开发？
A: 使用 pytest-xdist 并行执行，拆分慢速测试
